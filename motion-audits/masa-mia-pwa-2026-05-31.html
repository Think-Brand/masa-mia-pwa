Reconnaissance

La PWA es híbrida: tienda consumer (catálogo, carrito, pedidos del cliente), KDS operativo de cocina (alta frecuencia, Fabi/Alex usando 50–200 veces al día), y una capa de marca fuerte con Miga como mascota lúdica. Esto significa que un solo lente no aplica; cada zona necesita su propio criterio.

Lo que ya está bien:

  • Sistema de animaciones de Miga con 9 variantes (anim-bounce, anim-sway, anim-pop, etc.) — buena base de personalidad.
  • Uso consistente de active:scale-95 en botones — feedback táctil clásico de Emil.
  • Custom bezier en cart-bounce y badge-pop ya suavizados en el commit anterior (de 1.56 → 1.2 overshoot).
  • fade-up usado de forma moderada en confirmaciones y modales — no es uniform-fade-spam.

Lo que detecté que falta o necesita ajuste:

  • ~10+ motion gaps — modales, panels y banners que aparecen/desaparecen sin transición (snap-in feel).
  • Reduced-motion guard parcial — solo cubre 4 de 9 animaciones Miga, no cubre fade-up ni component-level.
  • Sin framer-motion — todo es CSS, lo cual está bien para mantener bundle bajo, pero limita la animación de exits (CSS no soporta enter+exit nativamente sin trucos).

#01

Reduced-motion guard parcial

Jakub · accesibilidad

src/app/globals.css:149–151 · cobertura: 4 de 9 animaciones Miga, 0 component-level

El guard actual solo cubre .anim-bounce, .anim-sway, .anim-float, .anim-jump. Quedan fuera: wiggle, shake, pop-in, breath, stir, todas las animaciones component-level (fade-up, Toast, BottomNav cart-bounce/badge-pop, catalogo qtyBump), y los keyframes inline de los modales.

Por qué es Critical: usuarios con vestibular sensitivity (más común de lo que se cree — ~35% de adultos reportan algún grado de sensibilidad al movimiento) reciben motion que pidieron explícitamente desactivar. La skill de Motion Principles lo marca como mandatory — no es opcional.

Fix: reemplazar el guard parcial por uno universal en globals.css que colapse cualquier animation y transition en el documento. Mantiene los final states intactos para que el layout no se rompa.

/* globals.css — al final del archivo */ @media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; animation-iteration-count: 1 !important; transition-duration: 0.01ms !important; scroll-behavior: auto !important; } }

#02

Modal CancelOrderModal aparece instantáneo

Jakub · production polish

src/app/mis-pedidos/page.tsx:306 · {cancelTarget && <CancelOrderModal />}

Conditional render sin envoltura de animación — el modal hace pop-in instantáneo cuando el usuario toca "Cancelar". Es UI crítica (acción destructiva), y el snap genera la sensación de "¿se abrió bien?" que rompe confianza.

Fix Jakub: enter con opacity + translateY + blur (la receta canónica de Jakub). El blur agrega esa sensación física de "materialización" que no logra un fade puro. Exit en mitad de intensidad — el usuario ya se está moviendo a otra cosa.

/* CancelOrderModal.tsx — wrapper interno */ @keyframes modalIn { 0% { opacity: 0; transform: translateY(16px) scale(0.96); filter: blur(4px); } 100% { opacity: 1; transform: translateY(0) scale(1); filter: blur(0); } } .modal-card { animation: modalIn 280ms cubic-bezier(0.22, 1, 0.36, 1) both; }

#03

Panel de NotificationBell aparece instantáneo

Jakub · origin-aware

src/components/NotificationBell.tsx:180 · {open && <panel />}

El panel de notificaciones (dropdown grande) aparece sin transición. Es UI consumer-facing, repetida (cliente lo abre cada vez que quiere ver el estado de su pedido), y la falta de animación lo hace sentir brusco, casi como un error del sistema.

Fix Jakub + Emil (origin-aware): el panel debe expandirse desde la campanita (top-right), no desde el centro. Emil insiste mucho en esto — los dropdowns deben respetar su trigger visual. transform-origin: top right + escala desde 0.97 + blur sutil.

/* NotificationBell.tsx — al panel */ .notif-panel { transform-origin: top right; animation: panelIn 220ms cubic-bezier(0.22, 1, 0.36, 1) both; } @keyframes panelIn { 0% { opacity: 0; transform: scale(0.97) translateY(-8px); filter: blur(3px); } 100% { opacity: 1; transform: scale(1) translateY(0); filter: blur(0); } }

#04

Toast staff usa keyframes — no se puede interrumpir

Emil · interruptibility

src/components/PedidosRealtime.tsx:148–155 · toast "Pedido nuevo: MM-XXXX"

El toast usa la clase fade-up que es @keyframes. Problema clásico de Emil: si llegan 2 pedidos seguidos (común en hora pico de un domingo), el segundo toast no puede interrumpir al primero porque keyframes no retarget mid-flight. El segundo se queda en cola, o pisa visualmente al primero sin gracia.

Fix Emil: usar CSS transition con clase de estado mounted en lugar de keyframes. Permite interrupción suave y retargeting. Plus: bajar duración a 200ms — es UI de alta frecuencia para Fabi/Alex, no necesita 400ms de fade.

/* Toast del staff — transition retargetable */ .staff-toast { opacity: 0; transform: translateY(-12px); transition: opacity 200ms cubic-bezier(0.16, 1, 0.3, 1), transform 200ms cubic-bezier(0.16, 1, 0.3, 1); } .staff-toast.mounted { opacity: 1; transform: translateY(0); }

#05

Banners de cortesía aparecen sin presentación

Jhey · momento de delight

src/app/carrito/page.tsx:332 (cumple) · :631 (welcome)

Cuando el usuario aplica un código de cumple o welcome courtesy, el banner aparece instantáneo. Este es exactamente el tipo de momento donde Jhey diría: "esto sí merece presentación". Es un delight moment intencional — el cliente acaba de recibir algo gratis.

Fix Jhey (con restraint): spring sutil con overshoot mínimo. Importante: este overshoot vive en un momento raro (aplicar cortesía pasa una vez por compra, no es high-frequency), así que cumple la frequency-gate de Emil para permitirse delight.

/* Banner cumple/welcome — entry spring */ @keyframes cortesia-pop { 0% { opacity: 0; transform: scale(0.95); } 50% { opacity: 1; transform: scale(1.03); } 100% { opacity: 1; transform: scale(1); } } .cortesia-banner { animation: cortesia-pop 420ms cubic-bezier(0.34, 1.3, 0.64, 1) both; transform-origin: center; }

#06

Miga loop infinito sin control de pausa

Jhey · accesibilidad vestibular

src/app/globals.css:127–135 · .anim-bounce, .anim-sway, .anim-float, .anim-jump, .anim-breath, .anim-stir

Las animaciones infinitas de Miga son personalidad de marca (Jhey las defendería siempre), pero hay un requisito de accesibilidad que la skill marca como obligatorio: "looping animations can be paused". El usuario debe poder pausar el movimiento ambiental aunque no tenga prefers-reduced-motion activado.

Fix mínimo: hacer las animaciones animation-play-state: paused al tap sostenido sobre Miga, o agregar un toggle global en /yo tipo "Miga tranquila". Lo primero es más juguetón (Jhey-friendly), lo segundo es más sobrio (Jakub-friendly).

Demo muestra la versión "tap para pausar" — Miga responde con un squash sutil que comunica el cambio de estado.

/* Miga con pausa al toque sostenido */ .miga-clickable { cursor: pointer; transition: transform 200ms ease; } .miga-clickable:active { transform: scale(0.94); animation-play-state: paused; } .miga-clickable:active * { animation-play-state: paused; }

#07

Errores de formulario con shake sutil

Emil · feedback claro

src/app/recuperar/[folio]/page.tsx:399, src/app/staff/login/page.tsx:123

Los mensajes de error aparecen instantáneos. Un shake de 250ms en el campo erróneo (no en el mensaje) ayudaría a localizar el problema sin gritar. La miga-shake que ya tienes en globals.css es casi perfecta — solo bájale duración a 250ms y aplícala al input, no al texto.

#08

Stagger ligero en confirmación de pedido

Jhey · momento ceremonial

src/app/confirmacion/[folio]/page.tsx — folio, contact_person, pickup_date, dirección

La pantalla de confirmación es un momento en la experiencia del cliente ("acabo de hacer mi pedido"). Las secciones podrían aparecer con un stagger muy ligero (60ms entre cada uno) — folio primero, contacto, fecha, dirección. Es la única pantalla que se ve una vez por compra, así que cumple la frequency-gate para delight. Cuidado: no lo apliques en /mis-pedidos ni en /catalogo, ahí sería stagger-spam.

#09

Tabs del catálogo sin transition al cambiar

Emil · clip-path

src/app/catalogo/page.tsx — tabs Todo / Roles / Berlinesas / RollinBox / LuvinBox

Al cambiar de tab, el contenido se reemplaza instantáneo. Emil recomienda clip-path para transiciones de tab — hardware-accelerated, sin layout shift, sin cambiar el DOM. Una transición de 180ms con clip-path + opacity le daría sensación de pulido sin sacrificar velocidad.

A través de cada lente

Through Jakub's lens

"Is this subtle and polished enough for production?"

El catálogo cliente está cerca pero le faltan transiciones en momentos clave (modales, panels, dropdowns). El stack CSS-puro es defendible — Jakub lo aprobaría sobre framer-motion en un proyecto con bundle restringido como una PWA — pero implica que los exits sean simétricos a los enters, y eso choca con su principio de "exits subtler than enters". Si crece la PWA, framer-motion con AnimatePresence sería el upgrade natural; mientras tanto, los enter-only son suficientes para Jakub. Score: 7/10 tras aplicar Critical 1–3.

Through Emil's lens

"Should this animate at all?"

El KDS de cocina se ve conservador en el lugar correcto — botones con scale-95, sin animaciones decorativas, urgencia comunicada por color (border-l-8 verde/amarillo/rojo en KitchenDisplay). Emil aprobaría. El único hallazgo es el Toast staff (Finding #4): keyframes no permite interrupción cuando llegan 2 pedidos seguidos. Fix con CSS transition + clase de estado. Adicionalmente, Emil aplaudiría quitar la cart-bounce del BottomNav si descubres en analítica que el cliente agrega productos 10+ veces por sesión — en ese punto cruza el threshold de "delight" a "ruido". Score: 8/10 para staff, 6.5/10 para cliente.

Through Jhey's lens

"What could this become?"

Miga es el alma de la PWA y eso se nota — 9 variantes de animación es generoso y honesto con la marca. Jhey defendería todas, pero también pediría dos cosas: (1) permitir pausarlas (Finding #6, no es opcional para accesibilidad), (2) premiar momentos raros con más cariño — la cortesía de cumple (Finding #5) y la confirmación de pedido (Opportunity #8) merecen celebración explícita, no son high-frequency. Aquí Jhey querría que se sienta el "te lo ganaste". Score: 9/10 en personalidad, 5/10 en accesibilidad — fixable.
